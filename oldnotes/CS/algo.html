Polynomial time

  • Is time taken to perform some operation where there are repetitive steps and the overall time required is a function of the number of steps to the power of some value. This is best explained using an example but the only ones I can come up with are examples of mathematical opertations. The simplest I can think of is tiling.
  • Suppose a task is to make a series of increasing squares from inch-square tiles and it takes 1 second to position each tile.
    • To do a 1 inch square will need 1 tile and this will take 1 second.
    • To do a 2 inch square will need 4 tiles and this will take 4 seconds
    • To do a 3 inch square will need 9 tiles and this will take 9 seconds.
    • To do a 4 inch square will need 16 tiles and this will take 16 seconds.
  • So the time required to lay out a square is polynomial according to the size of the square where:
  • T (seconds) = 1 second x square size in inches ^ 2

P vs NP

  • http://en.wikipedia.org/wiki/Complexity_classes_P_and_NP
  • http://stackoverflow.com/questions/111307/whats-p-np-and-why-is-it-such-a-famous-question
  • P class: The general class of questions for which some algorithm can provide an answer in polynomial time is called "class P"
  • NP class: For some questions, there is no known way to find an answer quickly, but if one is provided with information showing what the answer is, it may be possible to verify the answer quickly. The class of questions for which an answer can be verified in polynomial time is called NP
    • Note that NP problems can be solved in polynomial time on a non-deterministic turing machine.
  • Example
    • Does any subset of {−2, −3, 15, 14, 7, −10, ....} add up to zero. There is no algorithm to find such subset in polynomial time (there exist one on exponential time).
    • However, given the answer of (-2 -3 -10 15) one can quickly verify that the answer is valid.
  • Various types
    • P class: Problem can be solved & verified in polynomial time
    • NP complete: Problem may not be solved in polynomial time but can be verified in polynomial time
    • NP hard: Problem may not be solved & verified in polynomial time

Iterative approach

Lets say your array of letters looks like this: "ABCDEFGH". You have three indices (i, j, k) indicating which letters you are going to use for the current word, You start with:

A B C D E F G H
^ ^ ^
i j k
First you vary k, so the next step looks like that:

A B C D E F G H
^ ^   ^
i j   k
If you reached the end you go on and vary j and then k again.

A B C D E F G H
^   ^ ^
i   j k

A B C D E F G H
^   ^   ^
i   j   k
Once you j reached G you start also to vary i.

A B C D E F G H
  ^ ^ ^
  i j k

A B C D E F G H
  ^ ^   ^
  i j   k
...
Written in code this look something like that


void print_combinations(const char *string)
{
    int i, j, k;
    int len = strlen(string);

    for (i = 0; i < len - 2; i++)
    {
        for (j = i + 1; j < len - 1; j++)
        {
            for (k = j + 1; k < len; k++)
                printf("%c%c%c\n", string[i], string[j], string[k]);
        }
    }
}

Leader election

Token Ring Networks

  • LCR (LeLann-Chang-Roberts)
    • Comparison based algorithm
    • Each node sends its UID around the ring.
    • Each node compares the received UID with own UID
      • If input UID is greater, its passed further, else blocked
      • If the input UID is same as self, it means it completed one full circle and hence can elect self as leader.
    • Time/Msg complexity:  O(n), O(n^2)
  • HS (Hirschberg-Sinclair)
    • Comparison based algorithm
    • Tries to optimize on the no. of messages sent out on the ring
    • Assumes bi-directional ring
    • Each node sends its UID in both directions for upto x no. of nodes
      • If both the UIDs are received back, it continues in next phase.
    • Each node compares the received UID with its own UID
      • If input UID is greater, its passed further if there's more node or else sent back to the originator
      • If input UID is lesser, it's discarded
      • If input UID is equal, can elect self as leader.
    • Time/Msg complexity:  O(n), O(nlogn)
  • ML
  • Time/Msg complexity:  O(n), O(n^2)


Ethernet Networks

    Implementation of a single node with failover mechanism, without load balancing.

    electing:
       broadcast my state frequently
       if I receive a notification that someone else is elected, go to standby
       if I receive a notification that someone else is electing, and has an earlier start time (or if equal, smaller nodename), go to standby
       if the election interval is over:
          if I can claim the election in the DB
             go to elected
          otherwise
             go to electing
    standby:
       don't broadcast my state
       if I receive a notification the elected node died, go to electing
       periodically:
          if not (verify that the elected node is still elected via DB)
             go to electing
    elected:
       broadcast my state occasionally
       if I receive a noticiation someone else got elected, yield, go to standby
       if I recieve a notification someone else is electing, broadcast my state immediately
       periodically:
          if not (I can confirm my claim in the DB and update my heartbeat)
             go to electing
